# ASM Transformation Service — End-to-End Process Map

## Overview

This document describes the complete journey of an instrument file from the moment it is generated by a laboratory instrument to the final validated, integrity-verified ASM output. It is intended as a walkthrough for customers to fully understand how the service works.

```
│   ┌──────────────────────┐              ┌──────────────────────┐   │

│              │                                      │              │

```

### Entry Point 1: Convert Instrument File (Steps 1–5)

Use this when you have a raw instrument output file and need to produce validated ASM. This is the full pipeline described below.

### Entry Point 2: Validate Existing ASM File (Standalone)

Use this when you already have an ASM file — whether generated by your own converter, a third-party tool, or a previous conversion — and want to check if it meets Allotrope standards.

**Via Dashboard:**
- Navigate to "Validate ASM File" tab
- Upload your ASM JSON file
- View validation results (errors, warnings, metrics)
- Download PDF attestation report

**Via API:**
```
POST https://4ndgjn16zd.execute-api.us-east-1.amazonaws.com/prod/validate

{
    "asm_data": { ... },
    "validation_level": "comprehensive",
    "generate_report": true,
    "file_name": "my_asm_file.json"
}
```

**Response includes:**
- `valid`: true/false
- `errors`: array of compliance errors
- `warnings`: array of non-blocking issues
- `metrics`: technique, confidence, measurement count, structural checks
- `certification_report`: base64-encoded PDF (if `generate_report: true`)

**Key point**: DVaaS validates any ASM file regardless of how it was created. It does not need the original instrument file or an instrument config. It only checks the ASM structure against Allotrope standards.

## Entry Point 1: Convert Instrument File — Step-by-Step

### Step 0: One-Time Setup

Before converting any files, the customer completes a one-time setup per instrument.

| Task | Who | How | Frequency |
|------|-----|-----|-----------|
| Create Instrument Config | Customer | Dashboard "Instrument Config Creator" tab | Once per instrument |
| Register Custom Converter (if needed) | Customer or AWS | Dashboard "Converter Management" tab or API | Once per instrument |
| Approve Custom Converter | Reviewer | Dashboard "Converter Management" tab | Once per converter version |

**Instrument Config** is a JSON file containing instrument metadata:
```json
{
    "vendor": "NOVABIO_FLEX2",
    "instrument_type": "solution_analyzer",
    "manufacturer": "Nova Biomedical",
    "model": "BioProfile FLEX2",
    "file_format": "csv",
    "serial_number": "FLEX2-2024-001",
    "software_version": "6.2.1"
}
```

This file is reused for every conversion from that instrument. Create it once, use it forever.

### Step 2: Unified Converter (Intelligent Routing)

The Unified Converter receives the request and determines the best conversion path. It evaluates routes in priority order and uses the first one that succeeds.

#### Route A: Custom Converter Registry (Highest Priority)

**When**: Instrument config provides `vendor` + `model` that match an APPROVED converter in the registry.

**How it works:**
1. Unified Converter queries DynamoDB `CustomConverterRegistry` table
2. Filters by `vendor` + `model` + `status = APPROVED`
3. If found, calls the Custom Converter Service Lambda
4. Custom Converter Service loads the converter code from S3
5. Executes `convert(file_content)` in a sandboxed Lambda
6. Returns ASM output + field_mapping

**Includes**: Embedded custom converters (e.g., Nova FLEX2 built into the Unified Converter)

**Field mapping**: ✅ Yes — custom converters are required to emit `field_mapping` per the [Custom Converter Requirements](CUSTOM-CONVERTER-REQUIREMENTS.md)

**Example instruments**: Nova BioProfile FLEX2, Charles River EndoScan-V, any customer-submitted converter

#### Route B: Allotropy Library (Second Priority)

**When**: Route A has no matching converter, and the file format matches one of 31 allotropy-supported instruments.

**How it works:**
1. Unified Converter calls the Multi-Instrument Service
2. Multi-Instrument Service detects the vendor from file content patterns
3. Calls `allotrope_from_file()` with the detected vendor
4. Allotropy library parses the file using its built-in instrument parser
5. Returns ASM output

**Field mapping**: ⚠️ Not yet — allotropy is a third-party library that doesn't emit field_mapping. This is a planned enhancement where we will use the same field_mapping logic from custom converters to verify allotropy output.

**Example instruments**: Beckman Vi-CELL BLU/XR, Molecular Devices SoftMax Pro, PerkinElmer EnVision, Roche CEDEX BioHT, and 27 more

#### Route C: ATaaS / AI-Powered (Fallback)

**When**: Routes A and B both fail (no custom converter, allotropy doesn't recognize the format).

**How it works:**
1. Unified Converter calls the ATaaS service
2. ATaaS sends the file content to AWS Bedrock Claude 3.5 Sonnet
3. Claude analyzes the file format, identifies the instrument type, and generates ASM
4. Also generates a standalone Python converter for future reuse

**Field mapping**: ❌ No — AI-generated ASM does not include field_mapping

**Limitations**: Works best for simple file formats. Complex files (large XML, multi-sheet Excel) may timeout due to API Gateway 30-second limit.

**Example use case**: First-time conversion of an unknown instrument format

#### Route Priority Summary

```
1. Custom Converter Registry  →  Exact match by vendor + model (APPROVED)
2. Embedded Custom Converter  →  Built-in converters (e.g., Nova FLEX2)
3. Allotropy Library          →  31 supported instruments (pattern matching)
4. ATaaS (AI)                 →  Bedrock Claude fallback (simple formats)
5. Error                      →  All routes failed
```

### Step 3: Validation (DVaaS)

After conversion, the ASM output is validated by the Data Validation as a Service (DVaaS).

**What DVaaS checks:**

| Check | Description | Severity |
|-------|-------------|----------|
| Schema compliance | Does the ASM follow the Allotrope structure for this instrument type? | Error |
| Required fields | Are `$asm.manifest`, measurement documents, sample documents present? | Error |
| Measurement identifiers | Are they UUID v4 format? | Error |
| Timestamps | Are they ISO 8601 with timezone? | Error |
| Data source traceability | Are calculated values linked to source measurements? | Error |
| Unit recognition | Are measurement units standard Allotrope units? | Warning |
| Equipment metadata | Are serial number and software version present? | Warning |
| Device control | Are device type and detection type specified? | Warning |

**Validation levels:**

| Level | Use Case | Strictness |
|-------|----------|------------|
| Basic | Quick structural check | Low |
| Comprehensive | Production use | Medium — our files pass this |
| Certification | Regulatory submission | High — requires full equipment metadata |

**Output:**
- Validation status: VALID or INVALID
- Error count and details
- Warning count and details
- File metrics (technique, confidence, measurement count)
- PDF attestation report (downloadable)

**Important**: DVaaS validates the ASM *structure*. It does not compare against the source file — that is what Data Integrity Verification does (Step 4).

### Step 4: Data Integrity Verification (Work in Progress)

This step proves that every value from the source instrument file was preserved exactly in the ASM output. No rounding, no silent data loss, no transformation.

> **Status**: This capability is currently implemented for the Nova FLEX2 embedded converter only. It is being rolled out to all conversion routes. See the implementation status table below.

**How it works:**
- The converter emits a `field_mapping` array during conversion
- Each entry records: source field name, source value, ASM field name, ASM value, unit
- The dashboard compares `source_value` to `asm_value` for every entry
- Result: "100% Data Integrity Confirmed" or a list of mismatches

**Example field_mapping entry:**
```json
{
    "source_field": "pH",
    "source_value": 7.183,
    "asm_field": "pH",
    "asm_value": 7.183,
    "unit": "pH"
}
```

**What gets verified (Nova FLEX2 example — all 40 CSV columns):**

| Category | Example Fields |
|----------|---------------|
| Measurements | pH, PO2, PCO2, O2 Saturation, CO2 Saturation |
| Metabolites | Glucose, Lactate, Glutamine, Glutamate, NH4+, Na+, K+, Ca++ |
| Calculated values | pH @ Temp, PO2 @ Temp, PCO2 @ Temp, HCO3 |
| Instrument state | Vessel Temperature, Vessel Pressure, Sparging O2% |
| Operational data | Flow times, Pre-Dilution Multiplier, Dilution Ratio |
| Metadata | Lot numbers, Vessel ID, Batch ID, Cell Type, Tray Location |

**Why this matters for regulatory compliance:**
- FDA 21 CFR Part 11 requires proof that electronic records are accurate and unaltered
- EMA Annex 11 requires data integrity throughout the data lifecycle
- An auditor can inspect the integrity report and see field-by-field proof that no data was lost or changed

**Implementation status:**

| Conversion Route | Field Mapping | Status |
|-----------------|---------------|--------|
| Embedded Custom (Nova FLEX2) | ✅ All 40 CSV columns | **Implemented** — live today |
| Custom Converter (Registry) | Required per spec | **Spec complete** — converters must include `field_mapping` per [requirements doc](CUSTOM-CONVERTER-REQUIREMENTS.md), not yet enforced at runtime |
| Allotropy Library (31 instruments) | Planned | **Planned** — will reuse field_mapping logic from custom converters to verify allotropy output |
| ATaaS (AI) | Not planned | **Not planned** — AI-generated ASM is best-effort, not suitable for regulatory integrity claims |

**Rollout plan:**
1. ✅ Nova FLEX2 embedded converter (done)
2. Next: Enforce `field_mapping` in Custom Converter Registry at registration time
3. Next: Build verification wrapper for allotropy conversions using existing field_mapping logic
4. Future: Evaluate feasibility for AI-generated conversions

| Output | Format | Description |

| Validation Report | PDF | DVaaS attestation with certificate ID, status, errors, warnings |
| Integrity Report | Dashboard table | Field-by-field source-to-ASM comparison with match status |

## Frequently Asked Questions

**Q: What if my instrument is already supported by allotropy but I want field_mapping?**
A: Register a custom converter for that instrument. Custom converters take priority over allotropy (Route A before Route B), and they include field_mapping for data integrity verification.

**Q: Do I need a different instrument config for each file?**
A: No. One config per instrument. Reuse it for every file from that instrument.

**Q: What happens if the conversion fails on all routes?**
A: The API returns an error with details about why each route failed. Typically this means the instrument needs a custom converter.

**Q: Can I validate an ASM file without converting?**
A: Yes. This is a standalone capability. Use the "Validate ASM File" tab or call the DVaaS endpoint directly with your ASM JSON. You don't need the original instrument file or an instrument config. DVaaS validates any ASM file regardless of how it was created — by our service, your own converter, or a third-party tool.

**Q: What's the difference between DVaaS validation and Data Integrity Verification?**
A: DVaaS checks if the ASM is *structurally correct* (valid schema, required fields, proper format). Data Integrity Verification checks if the ASM is *faithful to the source* (every value preserved exactly). DVaaS only sees the ASM. Integrity verification compares the ASM against the original instrument file. Both are needed for full regulatory compliance.

**Q: Does Data Integrity Verification work for all instruments?**
A: Not yet. It is currently implemented for the Nova FLEX2 converter. Custom converters submitted through the registry are required to include field_mapping per our requirements spec. Allotropy-based conversions do not have it yet but it is planned. See the implementation status table in Step 4.

**Q: How long does conversion take?**
A: Custom converters and allotropy: < 1 second. ATaaS (AI): 5-30 seconds depending on file complexity.

**Q: Is my data stored?**
A: Files are processed in-memory only. No instrument data is persisted unless you explicitly request storage via the `store_results` flag.
