Generate Converter with AI

POST /generate-converter (ATaaS)

Analyzes a sample instrument file and generates a Python converter using AWS Bedrock Claude. The generated converter follows the Custom Converter Requirements and is auto-registered as PENDING.

Request

{
  "file_content": "<raw instrument file content>",
  "file_name": "SampleResults.csv",
  "manifest": {
    "vendor": "NOVABIO_FLEX2",
    "model": "BioProfile FLEX2",
    "instrument_type": "solution_analyzer",
    "file_format": "csv"
  },
  "auto_register": true
}

Response

{
  "converter_id": "novabio-flex2-bioprofile-flex2-ai-v1",
  "converter_code": "def convert(file_content): ...",
  "file_analysis": {
    "file_format": "csv",
    "instrument_type": "solution_analyzer",
    "total_rows": 27,
    "columns": "Date & Time,Sample ID,pH,..."
  },
  "registration": {
    "registered": true,
    "converter_id": "novabio-flex2-bioprofile-flex2-ai-v1",
    "status": "PENDING"
  },
  "status": "success"
}

Note: Only headers + first 3 rows are sent to AI. The generated converter handles all rows. Auto-registered converters require human approval before use.

Convert an Instrument File (Python)

import json
import requests

UNIFIED_ENDPOINT = "https://YOUR_UNIFIED_CONVERTER_ENDPOINT/convert"

# Read your instrument file
with open("SampleResults2025-November.csv", "r") as f:
    file_content = f.read()

# Read your instrument config
with open("novabio_flex2_config.json", "r") as f:
    config = json.load(f)

# Convert
response = requests.post(UNIFIED_ENDPOINT, json={
    "file_content": file_content,
    "file_name": "SampleResults2025-November.csv",
    "manifest": config
})

result = response.json()
print(f"Status: {result['status']}")
print(f"Method: {result['method']}")
print(f"Converter: {result['converter_used']}")

# Save ASM output
with open("output.asm.json", "w") as f:
    json.dump(result["asm_output"], f, indent=2)

# Check data integrity (if field_mapping available)
if result.get("field_mapping"):
    total = len(result["field_mapping"])
    matched = sum(1 for m in result["field_mapping"] if m["source_value"] == m["asm_value"])
    print(f"Data Integrity: {matched}/{total} fields matched ({matched/total*100:.0f}%)")

Validate an ASM File (Python)

import json
import base64
import requests

DVAAS_ENDPOINT = "https://YOUR_DVAAS_ENDPOINT/validate"

# Read your ASM file
with open("output.asm.json", "r") as f:
    asm_data = json.load(f)

# Validate
response = requests.post(DVAAS_ENDPOINT, json={
    "asm_data": asm_data,
    "validation_level": "comprehensive",
    "generate_report": True,
    "file_name": "output.asm.json"
})

result = response.json()
print(f"Valid: {result['valid']}")
print(f"Errors: {len(result.get('errors', []))}")
print(f"Warnings: {len(result.get('warnings', []))}")

# Print errors if any
for err in result.get("errors", []):
    print(f"  ERROR: {err}")

# Print metrics
metrics = result.get("metrics", {})
print(f"Technique: {metrics.get('technique', 'unknown')}")
print(f"Measurements: {metrics.get('measurement_count', 0)}")
print(f"Schema errors: {metrics.get('schema_errors', 'N/A')}")

# Save PDF report if available
if result.get("certification_report", {}).get("available"):
    pdf_bytes = base64.b64decode(result["certification_report"]["data"])
    with open("validation_report.pdf", "wb") as f:
        f.write(pdf_bytes)
    print("PDF report saved to validation_report.pdf")

End-to-End: Convert + Validate (Python)

import json
import base64
import requests

UNIFIED_ENDPOINT = "https://YOUR_UNIFIED_CONVERTER_ENDPOINT/convert"
DVAAS_ENDPOINT = "https://YOUR_DVAAS_ENDPOINT/validate"

# Step 1: Read files
with open("SampleResults2025-November.csv", "r") as f:
    file_content = f.read()
with open("novabio_flex2_config.json", "r") as f:
    config = json.load(f)

# Step 2: Convert
convert_resp = requests.post(UNIFIED_ENDPOINT, json={
    "file_content": file_content,
    "file_name": "SampleResults2025-November.csv",
    "manifest": config
}).json()

print(f"Conversion: {convert_resp['status']} via {convert_resp['method']}")

# Step 3: Validate
validate_resp = requests.post(DVAAS_ENDPOINT, json={
    "asm_data": convert_resp["asm_output"],
    "validation_level": "comprehensive",
    "generate_report": True,
    "file_name": "SampleResults2025-November.asm.json"
}).json()

print(f"Validation: {'VALID' if validate_resp['valid'] else 'INVALID'}")
print(f"Errors: {len(validate_resp.get('errors', []))}")
print(f"Warnings: {len(validate_resp.get('warnings', []))}")

# Step 4: Save outputs
with open("output.asm.json", "w") as f:
    json.dump(convert_resp["asm_output"], f, indent=2)

if validate_resp.get("certification_report", {}).get("available"):
    pdf_bytes = base64.b64decode(validate_resp["certification_report"]["data"])
    with open("validation_report.pdf", "wb") as f:
        f.write(pdf_bytes)

print("Done. Files saved: output.asm.json, validation_report.pdf")

Custom Converter Requirements

Function Signature

def convert(file_content):
    """
    Args:
        file_content (str): Raw file content as a string.
    Returns:
        dict: {
            "success": True/False,
            "asm_output": { ... },
            "field_mapping": [ ... ],
            "error": "message"  # on failure
        }
    """

Field Mapping Entry Format

{
  "source_field": "pH",
  "source_value": 7.183,
  "asm_field": "pH",
  "asm_value": 7.183,
  "unit": "pH"
}

Rules

  • Function must be named convert and accept a file_content string
  • field_mapping must cover every value from the source file
  • source_value must equal asm_value — no rounding or transformation
  • Measurement identifiers must be UUID v4
  • Timestamps must be ISO 8601 with timezone
  • No filesystem access, network calls, or dangerous functions (eval, exec, subprocess)
  • Must handle empty or malformed input gracefully

Validation Checklist

#Check
1Function named convert accepting file_content string
2Returns success/failure dict with asm_output and field_mapping
3field_mapping covers every source value
4source_value equals asm_value for every entry
5ASM includes $asm.manifest with correct URL
6UUID v4 measurement identifiers
7ISO 8601 timestamps with timezone
8No filesystem access
9No network calls
10No dangerous functions
11Handles bad input gracefully
12Tested with real instrument file

How Validation Works

DVaaS validates ASM files using a multi-layer approach based on official Allotrope standards. Understanding how validation works helps you interpret results and build compliant converters.

Validation Engine: jsonschema-rs

The core validation engine is jsonschema-rs, a Rust-based JSON Schema validator that supports JSON Schema draft 2020-12. This is the same standard used by the Allotrope Foundation to define their ASM schemas.

We chose jsonschema-rs because:

  • It implements the same JSON Schema draft (2020-12) that Allotrope uses
  • It handles complex schema features: $ref resolution, allOf/anyOf composition, pattern validation
  • It's fast — validates complex ASM files in milliseconds
  • It runs as a Lambda Layer alongside the DVaaS function

Allotrope JSON Schemas

The official Allotrope ASM schemas are published at gitlab.com/allotrope-public/asm. We bundle these schemas locally in the DVaaS Lambda deployment package under schemas/.

The schema hierarchy works as follows:

$asm.manifest (in your ASM file)
    → resolves to a manifest file (e.g., solution-analyzer.manifest)
    → manifest references a JSON schema (e.g., solution-analyzer.schema)
    → schema uses $ref to compose from:
        ├── core/hierarchy.schema  (common ASM structure)
        ├── core/core.schema       (base types, ordered items)
        └── detector schemas       (instrument-specific fields)
            ├── ph-detector.schema
            ├── blood-gas-detector.schema
            ├── metabolite-detector.schema
            ├── osmolality-detector.schema
            └── cell-counting-detector.schema

For example, a solution analyzer ASM file with manifest http://purl.allotrope.org/manifests/solution-analyzer/REC/2025/06/solution-analyzer.manifest is validated against the solution analyzer schema, which composes measurement validation from pH, blood gas, metabolite, osmolality, and other detector schemas.

Schema Resolution Process

  1. DVaaS reads the $asm.manifest value from your ASM file
  2. It looks up the corresponding manifest file in the local manifest registry
  3. The manifest points to the appropriate JSON schema
  4. If no manifest match is found, it derives the schema from the manifest URI pattern
  5. jsonschema-rs loads the schema and resolves all $ref references from the local schema registry
  6. The ASM file is validated against the fully resolved schema

Validation Layers

Validation runs in three layers:

LayerWhat It ChecksEngine
Layer 1: JSON Schema
  • ASM structure matches the Allotrope schema for the declared instrument type
  • Required fields are present (measurement identifier, measurement time, etc.)
  • Field types are correct (strings, numbers, arrays, objects)
  • Nested document structure follows the hierarchy schema
  • Measurement documents conform to the correct detector schema
 jsonschema-rs
Layer 2: $asm Attributes
  • Sample role types match Allotrope ontology terms
  • Container types match Allotrope ontology terms
  • Unit symbols are collected for reporting
 Custom Python
Layer 3: Supplementary
  • Calculated data has data source traceability (regulatory requirement)
  • Field names use spaces not hyphens (ASM convention)
  • Measurement identifiers are present
  • Technique detection from ASM structure
 Custom Python

Validation Levels

LevelBehaviorUse Case
Basic Runs all three layers. Reports errors and warnings separately. Quick check during development
Comprehensive Same as Basic. This is the recommended default. Production validation
Certification Strict mode — all warnings are promoted to errors. If the file passes with zero errors, an ALLOTROPE_CERTIFIED certificate is issued with a unique certificate ID. Regulatory submission, audit trail

Schema Updates

Allotrope publishes schema updates periodically. To update the schemas used by DVaaS:

  1. Download the latest schemas from gitlab.com/allotrope-public/asm
  2. Replace the contents of services/dvaas/schemas/json-schemas/ and services/dvaas/schemas/manifests/
  3. Redeploy DVaaS: cd services && cdk deploy

The schema registry is built dynamically at Lambda cold start — no code changes needed when schemas are updated.
