## Key Decisions Made

### 1. Architecture Choice
- **Decision**: Move from traditional service to AI agent-based architecture with **Supervisor Agent**
- **Reason**: Better for open source adoption, extensible, intelligent decision making, true modularity
- **Implementation**: Multi-agent system with intelligent supervisor pattern

### 🆕 8. **Autonomous Services Architecture**
- **Decision**: Split into two autonomous services that can run independently
- **ATaaS (ASM Transformation as a Service)**: Converts files AND generates standalone converter code
- **DVaaS (Data Validation as a Service)**: Validates ASM files against Allotrope schemas
- **Integration**: DVaaS used internally by ATaaS for quality assurance + available as standalone service
- **Benefits**: Microservices architecture, regulatory compliance, third-party integration

### 🆕 9. **GxP Compliance Requirements**
- **Decision**: Full GxP and federal guidelines compliance for pharmaceutical industry
- **Requirements**: Audit trails, electronic signatures, version control, validation documentation
- **Implementation**: Change control processes, data integrity controls, compliance reporting
- **Benefits**: Regulatory approval, pharmaceutical industry adoption

### 2. Supervisor Agent Pattern
- **Decision**: Implement true supervisor agent instead of static orchestrator
- **Benefits**: Dynamic workflow planning, runtime adaptation, load balancing, error recovery
- **Intelligence**: Uses Claude to make workflow decisions and handle failures

### 3. Knowledge Base Integration
- **Decision**: Use instrument documentation as knowledge base for deterministic converter generation
- **Reason**: Team suggestion to make converters more accurate and scientifically precise
- **Benefits**: Deterministic vs probabilistic generation, exact field mappings, proper calibrations
- **Implementation**: Knowledge Base Agent + Enhanced Converter Generation Agent

### 4. Strands Framework Integration
- **Decision**: Use Strands agents framework instead of custom agent implementation
- **Reason**: Industry best practices, proper tool integration, extended thinking capabilities
- **Benefits**: Better agent coordination, structured interfaces, enhanced reasoning
- **Implementation**: Refactored all agents to use Strands patterns with @tool decorators

### 5. Allotrope Foundation Integration
- **Decision**: Integrate with official Allotrope GitLab repository for standards compliance
- **Repository**: `https://gitlab.com/allotrope-public/asm.git`
- **Benefits**: Regulatory compliance, official templates, schema validation
- **Implementation**: Allotrope Standards Agent with daily synchronization

### 6. **🆕 Simple POC Deployment**
- **Decision**: Deploy minimal working POC for customer demonstration
- **Reason**: Customer meeting alignment - need working demo quickly
- **Benefits**: Proves AI-powered conversion concept, ready for testing
- **Implementation**: CDK stack with Lambda + API Gateway + Bedrock Claude

### 7. **🆕 AWS Greengrass Edge Client**
- **Decision**: Use AWS Greengrass for laboratory edge gateway client
- **Benefits**: Secure lab-to-cloud bridge, local processing, offline capability, GxP compliance
- **Implementation**: File watcher + secure uploader + audit logging at edge
- **Integration**: Calls existing API Gateway endpoints from laboratory environment

### 8. Technology Stack
- **AI Models**: AWS Bedrock Claude 3.7 Sonnet (via Strands BedrockModel + direct Lambda)
- **Agent Framework**: Strands Agents with strands-agents-tools (future full implementation)
- **Language**: Python for agents, TypeScript/Python for generated converters
- **Deployment**: **🚀 LIVE on AWS** - Container-based for future scaling
- **Knowledge Storage**: S3 bucket with instrument documentation (planned)

## 🎯 ARCHITECTURAL DECISION - Custom Converter Integration (January 21, 2026)

### Problem Statement
Complex files (Nova FLEX2, EndoScan-V, Wyatt ASTRA) require custom converters that are:
- ❌ Not in allotropy library (31 instruments)
- ❌ Too complex for ATaaS (timeout after 30 seconds)
- ✅ Manually written Python scripts (working, but standalone)
- ❓ Need integration into the system workflow

### Decision: Option 4 - Unified Converter Enhancement + Approval Workflow

**Architecture**:
```
User uploads file + manifest.json
    ↓
Unified Converter reads manifest (vendor, model, instrument_type)
    ↓
Routing Logic:
  1. Check custom converter registry (DynamoDB)
     → If found: Use custom converter
  2. Check allotropy support (31 instruments)
     → If found: Use Multi-Instrument service
  3. Check file complexity
     → If simple: Use ATaaS (AI-powered)
     → If complex: Return error (needs custom converter)
    ↓
All converters (custom + AI) → Approval Workflow
    ↓
Approved converters → Added to registry
    ↓
Validation via DVaaS
    ↓
Certified ASM output
```

### Components Needed

**1. Custom Converter Registry (DynamoDB)**
```
Table: CustomConverterRegistry
Fields:
  - converter_id (PK)
  - vendor (e.g., "novabio", "wyatt", "charles_river")
  - model (e.g., "flex2", "astra", "endoscan_v")
  - instrument_type (e.g., "solution_analyzer", "chromatography")
  - file_format (e.g., "csv", "xml")
  - converter_location (S3 path or Lambda ARN)
  - status (APPROVED, PENDING, DEPRECATED)
  - version
  - created_at
  - approved_by
```

**2. Custom Converter Service (Lambda)**
```python
# New service: Custom Converter Executor
# Loads and executes approved custom converters
# Input: converter_id, file_content
# Output: ASM JSON
```

**3. Enhanced Unified Converter**
```python
def route_conversion(file_content, manifest):
    vendor = manifest['vendor']
    model = manifest['model']
    
    # 1. Check custom converter registry
    custom_converter = lookup_custom_converter(vendor, model)
    if custom_converter and custom_converter['status'] == 'APPROVED':
        return execute_custom_converter(custom_converter, file_content)
    
    # 2. Check allotropy support
    if is_allotropy_supported(vendor, model):
        return call_multi_instrument(file_content)
    
    # 3. Fallback to ATaaS (simple files only)
    if is_simple_file(file_content):
        return call_ataas(file_content)
    
    # 4. No converter available
    return error("Complex file requires custom converter. Please submit for development.")
```

**4. Converter Deployment Workflow**
```
Developer creates custom converter (e.g., nova_flex2_converter.py)
    ↓
Submit to Approval Workflow
    ↓
Human reviewer validates:
  - Code quality
  - ASM compliance (via DVaaS)
  - Test with sample files
  - Security review
    ↓
Approve with electronic signature
    ↓
Deploy to Custom Converter Service
    ↓
Add entry to CustomConverterRegistry
    ↓
Available for use via Unified Converter
```

### Storage Options for Custom Converters

**Option A: S3 Storage (Recommended)**
- Store converter code in S3
- Custom Converter Service downloads and executes
- Pros: Easy deployment, versioning, no Lambda redeployment
- Cons: Security (executing uploaded code), need sandboxing

**Option B: Lambda Layers**
- Package converters as Lambda layers
- Attach to Custom Converter Service
- Pros: AWS-native, secure, versioned
- Cons: Requires redeployment for new converters

**Option C: Embedded in Lambda**
- Include converters in Lambda deployment package
- Pros: Simple, secure, fast
- Cons: Requires full redeployment for each new converter

**Decision**: Start with **Option C** (embedded), migrate to **Option B** (layers) as we scale

### Security Considerations

1. **Code Review**: All custom converters must pass human review
2. **Sandboxing**: Execute converters in isolated environment
3. **Input Validation**: Validate file content before passing to converter
4. **Output Validation**: Validate ASM output via DVaaS
5. **Access Control**: Only approved converters can be executed
6. **Audit Logging**: Log all converter executions
