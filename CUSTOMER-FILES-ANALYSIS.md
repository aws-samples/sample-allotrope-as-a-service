## Key Differences: Source CSV vs ASM JSON

| Aspect | Source CSV | Output ASM JSON |
|--------|-----------|-----------------|
| **Format** | Flat CSV (1 row = all measurements) | Hierarchical JSON (structured by measurement type) |
| **Standards** | Vendor-specific (Nova format) | Industry standard (Allotrope) |
| **Timestamps** | "11/1/2025 4:46:26 AM" | ISO8601: "2025-11-01T04:46:26.000+00:00" |
| **Metadata** | Minimal | Rich (device info, analyst, UUIDs, units) |
| **Interoperability** | Nova-specific | Works with any ASM-compliant system |
| **Structure** | 40 columns in single row | 4 nested measurement documents |
| **Traceability** | Limited | Full audit trail with identifiers |

## Validation Workflow

### **Scenario**: Customer sends ASM output files (without source CSV) for validation

### **API Call**:
```bash
POST https://4ndgjn16zd.execute-api.us-east-1.amazonaws.com/prod/validate
{
  "asm_data": { ... ASM JSON content ... },
  "validation_level": "comprehensive",  # or "basic" or "certification"
  "use_allotropy_validator": true
}
```

### **Validation Levels**:

#### **1. BASIC VALIDATION** (validation_level: "basic")
Quick structural checks:
- ✅ Valid JSON format
- ✅ Has `$asm.manifest` field
- ✅ Manifest URL is correct format (`http://purl.allotrope.org/manifests/...`)
- ✅ Has measurement documents
- ✅ Measurements are arrays
- ⚠️ Returns warnings for missing fields

**Use case**: Quick sanity check during development

#### **2. COMPREHENSIVE VALIDATION** (validation_level: "comprehensive")
Uses `validate_asm.py` (borrowed from Anthropic) to check:

**Structure Validation**:
- ✅ **Technique detection**: Detects "solution analyzer" from document structure
- ✅ **Naming conventions**: Checks for space-separated fields (not hyphens)
- ✅ **Measurement count**: Counts measurement documents
- ✅ **Nested structure**: Validates proper document nesting:
  - Sample fields in `sample document`
  - Device control fields in `device control aggregate document`
  - Custom fields in `custom information document`

**Data Validation**:
- ✅ **Sample roles**: Validates sample role types (standard, blank, control, etc.)
- ✅ **Units**: Validates units (mmHg, mmol/L, degC, pH, etc.)
- ✅ **Metadata**: Checks for device info, software name, timestamps
- ✅ **Calculated data traceability**: Checks for data source references
- ✅ **Unique identifiers**: Validates measurement identifiers exist

**Use case**: Pre-production validation, quality assurance

#### **3. CERTIFICATION VALIDATION** (validation_level: "certification")
- Same as comprehensive but **strict mode** (warnings become errors)
- Issues **ALLOTROPE_CERTIFIED** certificate if passes:

```json
{
  "valid": true,
  "certification": {
    "status": "ALLOTROPE_CERTIFIED",
    "certificate_id": "CERT-20260114182300",
    "issued_at": "2026-01-14T18:23:00Z",
    "validator": "allotropy_v1.1.1"
  }
}
```

**Use case**: Regulatory submission, FDA compliance, long-term archival

## What Validation DOES and DOES NOT Do

### **What Validation DOES Do** ✅

1. **Validates ASM compliance** - Follows Allotrope standards
2. **Validates structure** - Proper nesting, required fields
3. **Validates format** - Correct units, naming conventions, data types
4. **Validates traceability** - Identifiers, data sources, audit trail
5. **Issues certification** - For regulatory compliance (FDA, EMA)

### **What Validation DOES NOT Do** ❌

1. **Does NOT validate data accuracy** - Cannot verify if pH 7.183 is scientifically correct
2. **Does NOT check against source CSV** - No source file provided in validation-only scenario
3. **Does NOT verify instrument calibration** - Cannot check if instrument was properly calibrated
4. **Does NOT validate business logic** - Cannot check if glucose level is in normal range
5. **Does NOT verify sample identity** - Cannot confirm sample ID matches physical sample

**Analogy**: Validation is like **spell-check for data** - it ensures the structure is correct, but doesn't verify the content is scientifically accurate.
